# Agent Sync

This repo is the canonical configuration center for:

- `skills/`
- `MCP/servers/`

The sync layer is intentionally one-way: `repo -> clients`.

## Supported Clients

| Client | Skills Target | MCP Target | Notes |
| --- | --- | --- | --- |
| Codex | `~/.codex/skills` | `~/.codex/config.toml` | Manages `mcp_servers.<name>` and creates a system-skill marker for `.system/` skills |
| Gemini | `~/.gemini/antigravity/skills` | `~/.gemini/antigravity/mcp_config.json` | Uses `mcpServers` JSON layout |
| Claude | `~/.claude/skills` | `~/.claude/settings.json` | Does not touch `~/.claude.json` |
| Opencode | `~/.opencode/skills` | `~/.opencode/config.toml` | Also manages `skills.paths` entries for the synced skill root |
| Ollama | none in v1 | none in v1 | Stub only, reserved for a future adapter |

## Entry Points

| Playbook | Purpose |
| --- | --- |
| `ansible/playbooks/sync_agent_clients.yml` | Apply managed skills and MCP records into local clients |
| `ansible/playbooks/scan_agent_clients.yml` | Compare local client state with repo state and output drift/import findings |
| `python3 scripts/agent_sync/async_sync.py start` | Launch a background sync job for UI-triggered refresh actions |
| `python3 scripts/agent_sync/async_sync.py status` | Poll background sync status, log path, and exit code |

## Default Behavior

| Topic | Behavior |
| --- | --- |
| Direction | `repo -> clients` only |
| Skills sync | Copies directories, never symlinks |
| MCP sync | Renders client-native TOML or JSON config files |
| Managed scope | Only repo-managed skill roots and curated `MCP/servers/*.toml` entries |
| Mirror roots | `skills/workbuddy-marketplace/` and `skills/workbuddy-plugins/` are excluded by default |
| Secrets | Repo does not store inline secrets; scan reports local inline-secret findings but does not import them |
| Check mode | Supported with `ansible-playbook -C -D` |

## Main Variables

| Variable | Default | Purpose |
| --- | --- | --- |
| `agent_sync_clients` | `["codex","gemini","claude","opencode"]` | Clients to process |
| `agent_sync_scopes` | `["skills","mcp"]` | Limit to `skills`, `mcp`, or both |
| `agent_sync_include_mirrors` | `false` | Include mirror/cache skill roots if explicitly enabled |
| `agent_sync_target_root_overrides` | `{}` | Redirect writes to temp directories for safe tests |
| `agent_sync_report_path` | `""` | Optional JSON report output for scan runs |

## Examples

Dry-run all supported clients into temporary targets:

```bash
ANSIBLE_CONFIG=ansible/playbooks/ansible.cfg ansible-playbook -i localhost, -c local ansible/playbooks/sync_agent_clients.yml -C -D \
  -e '{"agent_sync_target_root_overrides":{"codex":"/tmp/ai-workflow-craft-sync/codex","gemini":"/tmp/ai-workflow-craft-sync/gemini","claude":"/tmp/ai-workflow-craft-sync/claude","opencode":"/tmp/ai-workflow-craft-sync/opencode"}}'
```

Sync only skills for Codex and Claude:

```bash
ANSIBLE_CONFIG=ansible/playbooks/ansible.cfg ansible-playbook -i localhost, -c local ansible/playbooks/sync_agent_clients.yml \
  -e '{"agent_sync_clients":["codex","claude"],"agent_sync_scopes":["skills"]}'
```

Start a background sync job suitable for a UI "Refresh / Sync" button:

```bash
python3 scripts/agent_sync/async_sync.py start --clients codex,claude --scopes skills
```

Poll the current background status:

```bash
python3 scripts/agent_sync/async_sync.py status
```

Scan local clients and write a report:

```bash
ANSIBLE_CONFIG=ansible/playbooks/ansible.cfg ansible-playbook -i localhost, -c local ansible/playbooks/scan_agent_clients.yml \
  -e '{"agent_sync_report_path":"/tmp/ai-workflow-craft-agent-scan.json"}'
```

## Implementation Notes

The sync layer is split into three pieces:

- normalize: `scripts/agent_sync/agent_sync.py`
- render: client-specific TOML and JSON output generated by the helper
- apply/scan: `ansible/roles/agent_sync/`

The helper rewrites managed config files after merging preserved local config with repo-managed MCP entries. Comments and original formatting are not preserved.

The async wrapper is intentionally thin: it does not change sync semantics, it only detaches execution, records a status JSON file, and writes a rolling log file under `~/.cache/ai-workflow-craft/agent-sync/`. A UI can safely wire this into:

- a manual "Refresh / Sync" button that calls `start`
- a polling loop that calls `status`
- a disabled-state guard while `running=true`
